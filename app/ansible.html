1. What is Ansible

Agentless automation tool using SSH and YAML for configuration, provisioning, and orchestration.

  • Push-based automation for quick change delivery.
  • No agents reduce operational overhead.
  • Standardizes deployments and avoids drift.

2. Architecture

Control node runs playbooks on managed nodes through SSH using modules, plugins, and inventory.

  • Managed nodes need only SSH + Python.
  • Plugins extend logging, caching, lookups.
  • Simple architecture scales easily across fleets.

3. Inventory

Defines host groups, variables, and target systems for automation workflows.

  • Supports INI, YAML, JSON formats.
  • Separates dev/stage/prod logically.
  • Stores host/group-specific variables.

4. Static vs Dynamic Inventory

Static = manually defined files; dynamic = cloud APIs for auto-discovery.

  • Static suitable for small infra.
  • Dynamic required for AWS/GCP autoscaling.
  • Supports filters and tag-based grouping.

5. YAML Essentials

Ansible playbooks use YAML for clean, structured, indentation-sensitive configuration.

  • Uses lists + key-value mappings.
  • No tabs allowed—spaces only.
  • Declarative and human-friendly syntax.

6. Playbooks

Define desired state and orchestrate tasks across hosts in predictable order.

  • Support variables, handlers, templates.
  • Readable and reusable structure.
  • Core execution unit in automation.

7. Tasks

Each task calls a module to enforce a specific state (package installed, file copied, etc.).

  • Executed sequentially within plays.
  • Use conditionals, loops, tags.
  • Idempotency ensures safe repetition.

8. Modules

Reusable units like package, file, user, service. Thousands available including cloud and network.

  • Return structured JSON results.
  • Support idempotent behavior.
  • Custom modules extend functionality.

9. Collections

Bundles of modules, roles, plugins packaged for reuse and versioning.

  • Installed via Ansible Galaxy.
  • Support version pinning.
  • Used for internal automation standards.

10. Roles

Structured reusable automation containing tasks, vars, handlers, templates, defaults.

  • Enforces separation of concerns.
  • Shareable through Galaxy or git repos.
  • Ideal for repeated configuration patterns.

11. Variables

Inject dynamic values into playbooks, templates, and modules.

  • Support lists, dicts, strings, booleans.
  • Defined at host, group, role, or play level.
  • Improve reuse across environments.

12. Variable Precedence

Defines which variable wins when defined in multiple places.

  • CLI vars override everything.
  • Role defaults have lowest priority.
  • Crucial for debugging unexpected behavior.

13. Templates (Jinja2)

Generate dynamic config files using Jinja2 templating features.

  • Supports loops, filters, and conditionals.
  • Used for service configuration files.
  • Integrates with variables and facts.

14. Handlers

Tasks that run only when notified, usually to restart or reload services.

  • Prevents unnecessary restarts.
  • Executed at end of play by default.
  • Ensures controlled state changes.

15. Conditionals

Control task execution based on facts, variables, or outputs.

  • Use when, failed_when, changed_when.
  • Supports multi-OS automation logic.
  • Avoids unnecessary task execution.

16. Loops

Repeat tasks over lists, dictionaries, or sequences to avoid duplication.

  • Supports loop, with_items, with_dict, etc.
  • Simplifies repetitive configuration tasks.
  • Useful in file creation, user mgmt, and service rules.

17. Tags

Selective execution mechanism for running only specific tasks or sections.

  • Useful for debugging or partial deployments.
  • Supports include, skip, and filtering.
  • Essential in CI pipelines for fast feedback loops.

18. Fact Gathering

Collects system information (OS, CPU, memory, networking) before execution.

  • Stored under ansible_facts.* variables.
  • Can be disabled to improve speed.
  • Supports custom facts for organization-specific data.

19. Filters

Transform or format variables inside templates and tasks using Jinja2 filters.

  • Common filters: default, replace, regex, join.
  • Enable clean template logic.
  • Extensible through custom filter plugins.

20. Lookup Plugins

Pull external data dynamically from files, environment, credentials, or APIs.

  • Executes on control node, not targets.
  • Useful for secrets, file reads, environment values.
  • Supports plugins like file, env, password.

21. Ansible Vault

Encrypt files or variables to protect secrets in playbooks and repos.

  • Supports multiple vault IDs for different environments.
  • Integrates with CI/CD securely.
  • Enables safe storage of credentials and keys.

22. SSH & Connections

The default connection method for communicating with managed nodes.

  • Supports SSH multiplexing and ControlPersist.
  • Works with bastion/jump hosts.
  • Alternatives: local, paramiko, network_cli.

23. Privilege Escalation

Use “become” to execute tasks as a different user (typically root).

  • Supports sudo, su, pbrun, doas.
  • Configurable at task, play, or global level.
  • Critical for package installs and OS changes.

24. Error Handling

Manage and control failures in Ansible runs for safer automation.

  • block/rescue/always enable try-catch-like behavior.
  • ignore_errors for non-critical tasks.
  • failed_when allows custom fail logic.

25. Check Mode

Dry-run mode that shows what changes would occur without applying them.

  • Useful for drift detection.
  • Not all modules fully support check mode.
  • Safe validation step before production runs.

26. Idempotency

Ensures repeated runs do not change system state unnecessarily.

  • Prevents drift across environments.
  • Makes automation predictable and stable.
  • Most core modules enforce idempotency.

27. Includes & Imports

Split automation into modular parts; include = dynamic, import = static.

  • Improves readability and reuse.
  • Supports conditional includes.
  • Works with tasks, plays, and roles.

28. ansible.cfg Optimization

Control Ansible's global behavior and performance tuning.

  • Enable pipelining and SSH multiplexing.
  • Increase forks for parallel execution.
  • Configure inventory paths and timeouts.

29. Strategy Plugins

Define how tasks are executed across multiple hosts.

  • linear: default sequential execution.
  • free: hosts run independently.
  • debug: step-by-step debugging mode.

30. Serial & Rolling Updates

Apply updates in controlled batches to avoid downtime.

  • serial controls batch size.
  • max_fail_percentage manages risk tolerance.
  • Essential for production patching strategies.

31. Dynamic Inventory (AWS)

Automatically discovers hosts from EC2 using tags and filters.

  • Eliminates manual host updates.
  • Supports autoscaling environments.
  • Integrates with AWS APIs for live inventory.

32. OS Patching Automation

Automates package updates and reboots across server fleets.

  • Use serial batches for safe rollouts.
  • Combine with “wait_for” for reboot handling.
  • Integrates with maintenance windows.

33. Kubernetes Automation

Manage k8s objects, Helm charts, and CRDs directly from Ansible.

  • Useful for cluster bootstrapping.
  • Automates add-ons and deployments.
  • Works well with GitOps workflows.

34. Cloud Provisioning

Provision small-scale AWS/Azure/GCP resources using modules.

  • Terraform preferred for large infra.
  • Ansible ideal for post-provisioning configuration.
  • Great for hybrid workflows.

35. Network Automation

Automates routers, switches, load balancers, and firewalls.

  • Supports Cisco, Juniper, Arista, F5.
  • Reduces manual CLI errors.
  • Works with network_cli & netconf connections.

36. Security Hardening

Automate OS-level hardening and compliance requirements using CIS or custom benchmarks.

  • Manage file permissions, firewall rules, SSH configs.
  • Eliminate manual configuration drift.
  • Ideal for baseline enforcement across fleets.

37. Compliance Automation

Validate and enforce organization-wide compliance policies automatically.

  • Implements remediation and reporting steps.
  • Supports audits and security governance.
  • Repeatable and version-controlled enforcement.

38. Molecule Testing

Framework for testing Ansible roles using containers or VMs before production use.

  • Ensures roles behave as expected.
  • Supports multiple drivers (Docker, Vagrant).
  • Critical for CI-based validation pipelines.

39. Linting & CI Validation

Static analysis and best-practice checks for playbooks and roles.

  • Use ansible-lint for consistent style.
  • Integrate into GitHub Actions/Jenkins.
  • Prevents poor-quality automation from merging.

40. GitOps with Ansible

Trigger automation based on Git events and version-controlled state.

  • AWX webhooks for CI/CD workflows.
  • Enables infra-as-pull-request workflows.
  • Keeps automation aligned with repo history.

41. Ansible AWX Basics

Web UI + API platform for scheduling, RBAC, credentials, and workflow management.

  • Central visibility of automation runs.
  • Secure storage of credentials.
  • Role-based execution controls.

42. Job Templates

Reusable execution definitions combining playbooks, credentials, and inventory.

  • Allow parameterized or scheduled runs.
  • Standardize deployments and patch cycles.
  • Support survey prompts for user inputs.

43. RBAC in AWX

Fine-grained access control to manage who can run, edit, or view automation.

  • Separate operator vs admin permissions.
  • Supports organization, team, and project roles.
  • Mandatory for production governance.

44. Webhooks (CI/CD)

Automatically trigger playbook runs from GitHub/GitLab push or merge events.

  • Enforces GitOps workflows.
  • Keeps infra in sync with code.
  • Reduces manual execution overhead.

45. Performance Tuning

Optimize Ansible for speed and scalability during large fleet automation.

  • Enable pipelining for fewer SSH ops.
  • Raise forks for parallel execution.
  • Disable unnecessary fact-gathering.

46. Inventory Caching

Store dynamic inventory results temporarily to avoid repeated API calls.

  • Boosts speed with AWS/GCP large-scale fleets.
  • Configurable via cache plugins.
  • Reduces load on cloud APIs.

47. Rolling Deployments

Deploy applications with controlled host-by-host rollout to avoid downtime.

  • Combine with serial batches.
  • Use health checks and wait conditions.
  • Ensures smooth production releases.

48. Blue-Green Automation

Parallel environments allow instant cutover between old and new deployments.

  • Zero-downtime release strategy.
  • Rollback-ready architecture.
  • Useful for apps and infra changes.

49. Orchestration Patterns

Coordinate multi-step workflows across apps, services, and nodes.

  • Use blocks, dependencies, and inventories.
  • Enables infra → config → app pipelines.
  • Foundation for full-stack automation.

50. Production Troubleshooting

Debug complex playbook failures under real production conditions.

  • Use -vvv logs for full visibility.
  • Validate variables, paths, and SSH.
  • Check module return JSON for clues.
